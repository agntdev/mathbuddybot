# fix-461586f4102f121a — Three different `formatNumber` implementations diverge in output

**Weight:** 0.0000 (share of project budget)
**Reward:** 0 MATHB

Four handlers use three different `formatNumber` functions with different behavior:

| Handler | Source | Behavior |
|---|---|---|
| `E1T1.ts` (add) | Local `toLocaleString("en-US")` | Adds thousand-separator commas |
| `E1T2.ts` (sub), `E1T4.ts` (div) | `toolkit/format.ts` via `Intl.NumberFormat("en-US", {maxFractionDigits:20})` | Adds thousand-separator commas, up to 20 fraction digits |
| `E1T3.ts` (mul) | `util/formatNumber.ts` via `toFixed(10)` + trailing-zero trim | No commas, max 10 fraction digits |

The spec (`docs/spec.md`) describes the `util/formatNumber.ts` behavior: *"up to 10 decimal places and trim trailing zeros"* with no commas. The toolkit's `format.ts` adds commas and uses 20 fraction digits, which contradicts the spec. `E1T1`'s local function adds commas via `toLocaleString`. For large inputs like `/sub 2000 1000`, the output would be `"2,000 - 1,000 = 1,000"` on add/sub/div but `"Result: 2000"` on mul — inconsistent and spec-violating. The handlers should converge on one shared formatter, preferably `util/formatNumber.ts`.
